## 1. 引言

### 1.1 目的

本文档旨在描述网络缓存管理客户端系统的整体架构设计、模块划分、技术选型及模块间的交互关系，为后续详细设计与开发提供指导。

### 1.2 适用范围

本文档适用于第四组全体开发人员，以及需要与本客户端联调的第二组（Linux 缓存服务端）开发人员。

### 1.3 术语表

| 术语 | 说明 |
|------|------|
| Cache Entry | 缓存条目，包含 key、value、TTL、创建时间 |
| TTL | Time To Live，缓存过期时间（秒） |
| Mock | 模拟实现，用于界面先行开发 |
| RESP | REdis Serialization Protocol，与服务端通信的文本协议 |

## 2. 总体架构

### 2.1 系统定位

本系统为**实习题目 2 —— 网络缓存管理客户端**，通过 **RESP 协议**（Redis 序列化协议）与**题目 1（Linux 缓存服务端）**通信，为用户提供图形化的缓存数据管理能力。

### 2.2 架构风格

采用**分层架构**，自顶向下分为三层：

```
┌─────────────────────────────────────────────┐
│              界面层 (UI Layer)               │
│    JavaFX + FXML                             │
│    MainView.fxml / MainController            │
├─────────────────────────────────────────────┤
│              接口层 (Interface Layer)         │
│    CacheServerClient (接口契约)              │
├─────────────────────────────────────────────┤
│              实现层 (Implementation Layer)    │
│    ┌─────────────────┐ ┌─────────────────┐  │
│    │  MockCacheClient │ │  RespCacheClient │  │
│    │  (开发调试用)     │ │  (联调/生产用)   │  │
│    └─────────────────┘ └─────────────────┘  │
└─────────────────────────────────────────────┘
```

### 2.3 设计原则

- **接口隔离**：界面层仅依赖 `CacheServerClient` 接口，不依赖具体实现
- **可替换性**：通过 `MockCacheClient` / `RespCacheClient` 切换，支持界面先行开发
- **配置驱动**：通过 `config.properties` 的 `client.mock` 项选择运行模式

## 3. 模块划分

### 3.1 模块列表

| 模块 | 包路径 | 职责 | 优先级 |
|------|--------|------|--------|
| 数据模型 | `model` | CacheEntry 实体定义（含 STRING/LIST 双类型） | P0 |
| 服务接口 | `net` | CacheServerClient 接口契约 | P0 |
| Mock实现 | `net` | 本地模拟缓存行为（ConcurrentHashMap） | P0 |
| RESP实现 | `net` | 与 Linux 服务端的 RESP 协议通信 | P1 |
| 协议工具 | `net/protocol` | RespCodec + RespResponse 编解码 | P1 |
| UI控制器 | `ui` | JavaFX 界面事件处理 | P0 |
| FXML视图 | `resources/...` | 界面布局定义 | P0 |
| 工具类 | `util` | 配置读取、数据导出等 | P1 |
| 测试 | `test/...` | 单元测试 | P0 |

### 3.2 模块依赖关系

```
ui (Controller)
  ↓ 依赖接口
net (CacheServerClient 接口)
  ↓ 实现               ↓ 实现
MockCacheClient       RespCacheClient
  ↓ 依赖                 ↓ 依赖
model (CacheEntry)    model (CacheEntry)
                          ↓ 依赖
                       protocol (RespCodec)
```

## 6. 通信协议

### 6.1 传输层

- **协议**：RESP（REdis Serialization Protocol），文本协议
- **传输**：TCP
- **默认地址**：`127.0.0.1:6379`
- **兼容性**：可使用 `redis-cli` 或 `telnet` 直接测试

### 6.2 消息格式

**请求格式（客户端发送 RESP 数组）：**

```
*参数个数\r\n
$第1参数长度\r\n
第1参数\r\n
$第2参数长度\r\n
第2参数\r\n
...
```

示例：`SET name Alice`
```
*3\r\n$3\r\nSET\r\n$4\r\nname\r\n$5\r\nAlice\r\n
```

**响应格式（服务端返回）：**

| 首字节 | 类型 | 示例 |
|:------:|------|------|
| `+` | 简单字符串 | `+OK\r\n` |
| `-` | 错误 | `-ERR unknown command\r\n` |
| `:` | 整数 | `:3\r\n` |
| `$` | 批量字符串 | `$5\r\nhello\r\n` |
| `$` | 空值 | `$-1\r\n` |
| `*` | 数组 | `*2\r\n$3\r\nGET\r\n$3\r\nkey\r\n` |

### 6.3 命令映射

| 客户端方法 | 服务端命令 | 请求示例 |
|-----------|-----------|---------|
| `ping()` | PING | `*1\r\n$4\r\nPING\r\n` |
| `get(key)` | GET | `*2\r\n$3\r\nGET\r\n$3\r\nkey\r\n` |
| `set(k, v)` | SET | `*3\r\n$3\r\nSET\r\n$1\r\nk\r\n$1\r\nv\r\n` |
| `set(k, v, ex)` | SET EX | `*5\r\n$3\r\nSET\r\n...\r\n$2\r\nEX\r\n$2\r\n60\r\n` |
| `del(k...)` | DEL | `*3\r\n$3\r\nDEL\r\n...` |
| `ttl(k)` | TTL | `*2\r\n$3\r\nTTL\r\n...` |
| `lpush(k, v...)` | LPUSH | `*4\r\n$5\r\nLPUSH\r\n...` |
| `rpush(k, v...)` | RPUSH | `*4\r\n$5\r\nRPUSH\r\n...` |
| `lpop(k)` | LPOP | `*2\r\n$4\r\nLPOP\r\n...` |
| `lrange(k, s, e)` | LRANGE | `*4\r\n$6\r\nLRANGE\r\n...` |

## 7. 非功能需求

### 7.1 性能

- 界面操作响应 < 100ms（Mock 模式下）
- 支持管理 10 万级别缓存条目

### 7.2 可用性

- 连接中断时显示明确状态提示
- 支持 Mock / RESP 双模式一键切换（配置化）

### 7.3 可维护性

- `CacheServerClient` 接口文档完整
- RESP 实现层独立可替换
- 协议编解码与通信逻辑分离（RespCodec 独立于 RespCacheClient）
