## 1 引言

### 1.1 目的

本文档为**网络缓存管理系统（Network Cache Management System）** 客户端的详细设计说明书，旨在描述系统的模块内部实现细节、数据结构定义、函数接口及算法流程，以指导开发人员编码实现和后续的测试活动。

本文档面向的开发范围包括：缓存客户端接口（CacheServerClient）、Mock客户端（MockCacheClient）、RESP协议编解码（RespCodec）、RESP客户端实现（RespCacheClient）、JavaFX界面控制器（MainController / TabPaneController / CacheEntryController / ConnectController）、数据导出工具（ExportUtil）及配置管理（Config）。

### 1.2 范围

本文档覆盖以下模块的详细设计：

- 缓存客户端接口模块（CacheServerClient）
- Mock客户端模块（MockCacheClient）
- RESP协议编解码模块（RespCodec）
- RESP响应模型模块（RespResponse）
- RESP客户端模块（RespCacheClient）
- 主界面控制器模块（MainController）
- 多客户端标签页模块（TabPaneController）
- 对话框控制器模块（CacheEntryController / ConnectController）
- 缓存条目加载模块（CacheEntryLoader）
- Key 通配符匹配模块（KeyPatternMatcher）
- 数据导出模块（ExportUtil）
- 配置模块（Config）

本文档不涉及服务端（第二组）的内部实现细节，仅描述客户端与服务端之间的通信接口和协议。

## 2 总体设计

### 2.1 系统架构

系统采用**分层架构设计**，自顶向下分为界面层（UI Layer）、接口抽象层（Interface Layer）和实现层（Implementation Layer），另设独立的协议层处理网络通信。各层职责明确，下层为上层提供原子操作服务。

```
┌──────────────────────────────────────────────┐
│  界面层（UI Layer）                            │
│  MainController / TabPaneController           │
│  CacheEntryController / ConnectController      │
│  JavaFX + FXML                                │
├──────────────────────────────────────────────┤
│  接口抽象层（Interface Layer）                   │
│  CacheServerClient 接口                        │
│  （connect/disconnect/isConnected/get/set/del/scan/ │
│   lpush/rpush/lpop/lrange/ping/ttl/reconnect/     │
│   getLastHost/getLastPort）                        │
├──────────────────────────────────────────────┤
│  实现层（Implementation Layer）                 │
│  ┌──────────────┐  ┌──────────────────┐      │
│  │MockCacheClient│  │RespCacheClient  │      │
│  │（本地内存模拟） │  │（RESP TCP通信）  │      │
│  └──────────────┘  └──────────────────┘      │
├──────────────────────────────────────────────┤
│  协议层（Protocol Layer）                      │
│  RespCodec（RESP编解码）                        │
│  RespResponse（RESP响应模型）                   │
└──────────────────────────────────────────────┘
```

**各层职责：**

- **界面层**：使用 JavaFX + FXML 构建图形用户界面，通过 `CacheServerClient` 接口与后端通信，不依赖具体实现。
- **接口抽象层**：定义缓存操作契约，UI 层只依赖此接口，支持 Mock / RESP 双模式切换。
- **实现层**：
  - `MockCacheClient` — 本地 ConcurrentHashMap 内存模拟，用于 UI 先行开发和调试，不依赖服务端。
  - `RespCacheClient` — RESP 协议 TCP 实现，用于对接第二组 Linux 缓存服务端。
- **协议层**：`RespCodec` 负责 RESP 协议请求编码和响应解码，`RespResponse` 封装 RESP 五种数据类型。

### 3.2 Mock客户端模块（MockCacheClient）

**文件路径：** `src/main/java/com/cache/client/net/MockCacheClient.java`

#### 3.2.1 数据结构定义

| 数据成员 | 类型 | 说明 |
|---|---|---|
| `stringStore` | `Map<String, CacheEntry>` | STRING 类型数据存储（ConcurrentHashMap） |
| `listStore` | `Map<String, List<String>>` | LIST 类型数据存储（ConcurrentHashMap） |
| `allKeys` | `Set<String>` | 所有 key 集合，用于 SCAN 模拟 |
| `connected` | boolean | 连接状态标记 |

#### 3.2.2 预置演示数据

| Key | 类型 | Value | TTL |
|---|---|---|---|
| user:1001 | STRING | Alice | 300s |
| user:1002 | STRING | Bob | 120s |
| session:abc123 | STRING | {"token":"xyz"} | 3600s |
| config:theme | STRING | dark | 永不过期 |
| rate:limit | STRING | 1000 | 60s |
| queue:tasks | LIST | [task1, task2, task3] | 永不过期 |
| queue:logs | LIST | [log-a, log-b, log-c, log-d] | 永不过期 |

#### 3.2.3 核心算法

**过期检查流程**（以 `get` 为例）：

1. 调用 `getStringIfValid(key)` 从 stringStore 获取 CacheEntry
2. 调用 `entry.isExpired()` 判断是否过期
3. 若已过期：从 stringStore 和 allKeys 中移除，返回 `Optional.empty()`
4. 若未过期：返回 `Optional.of(entry.getValue())`
5. LIST 类型键不可用 GET 访问，返回 `Optional.empty()`

**通配符匹配算法（`match` 方法）：**

将包含 `*` 和 `?` 的 pattern 转换为 Java 正则表达式：
- `*` → `.*`（匹配任意字符序列）
- `?` → `.`（匹配单个字符）
- 使用 `String.matches()` 进行正则匹配

**类型互转规则（Mock 特有）：**
- `SET` 覆盖 LIST 类型 key：删除 listStore 中对应条目
- `LPUSH` / `RPUSH` 覆盖 STRING 类型 key：删除 stringStore 中对应条目

### 3.3 RESP协议编解码模块（RespCodec）

**文件路径：** `src/main/java/com/cache/client/net/protocol/RespCodec.java`

#### 3.3.1 数据结构定义

本模块为静态工具类，无实例数据成员。所有方法均为 `static`。

#### 3.3.2 核心方法

| 方法 | 说明 |
|---|---|
| `encode(args)` | 将命令参数编码为 RESP 数组格式字节数组。格式：`*N\r\n$len\r\narg\r\n...` |
| `decode(in)` | 从 InputStream 读取并解析一个 RESP 响应。支持五种类型标记：`+ - : $ *` |
| `encodeToString(args)` | 编码并返回字符串形式（用于日志/调试） |

#### 3.3.3 请求编码格式

```
输入: ("SET", "key", "value")
输出: *3\r\n$3\r\nSET\r\n$3\r\nkey\r\n$5\r\nvalue\r\n
```

编码流程：
1. 写入数组头部 `*N\r\n`（N 为参数个数）
2. 对每个参数，依次写入 `$len\r\n` 和参数字节数据 + `\r\n`
3. 参数长度按 UTF-8 字节数计算

#### 3.3.4 响应解码流程

1. 读取首字节，判断 RESP 类型标记符：
   - `+`（0x2B）→ **简单字符串（Simple String）**：读取一行直到 `\r\n`
   - `-`（0x2D）→ **错误（Error）**：读取一行
   - `:`（0x3A）→ **整数（Integer）**：读取一行，解析为 long
   - `$`（0x24）→ **批量字符串（Bulk String）**：读取长度，-1 表示 null，否则读取 N 字节 + `\r\n`
   - `*`（0x2A）→ **数组（Array）**：读取元素个数，递归读取每个元素
2. 返回对应的 `RespResponse` 实例

#### 3.3.5 流读取辅助方法

| 方法 | 说明 |
|---|---|
| `readLine(in)` | 逐字节读取直到遇到 `\r\n`，返回字符串（不含 `\r\n`） |
| `readExact(in, len)` | 从流中精确读取 len 字节（循环 read 直到读取完毕） |
| `expectCrLf(in)` | 验证接下来的两个字节是 `\r\n` |

### 3.5 RESP客户端模块（RespCacheClient）

**文件路径：** `src/main/java/com/cache/client/net/RespCacheClient.java`

#### 3.5.1 数据结构定义

| 成员 | 类型 | 说明 |
|---|---|---|
| `socket` | Socket | TCP Socket 连接 |
| `in` | InputStream | Socket 输入流 |
| `out` | OutputStream | Socket 输出流 |
| `connected` | boolean | 连接状态标记（volatile，多线程可见） |
| `lastHost` | String | 上次成功连接的地址，用于自动重连 |
| `lastPort` | int | 上次成功连接的端口 |
| `lock` | Object | 串行化 socket 操作的同步锁 |

#### 3.5.2 核心方法

| 方法 | 说明 |
|---|---|
| `execute(args)` | 通用方法：编码并发送 RESP 请求，解码返回 RespResponse；未连接时抛出明确提示 |
| `expectOk(args)` | 发送命令，期望返回 `+OK`，失败抛异常 |
| `expectInteger(args)` | 发送命令，返回整数值 |
| `expectBulkString(args)` | 发送命令，返回批量字符串（可能为 null） |
| `expectArray(args)` | 发送命令，返回字符串数组 |
| `reconnect()` | 断开旧连接并重新连接到 `lastHost:lastPort`，返回 boolean |

#### 3.5.3 通信流程

每个命令通过 `execute()` 完成"编码 → 发送 → 解码"三步操作。`execute()` 通过 `synchronized(lock)` 保证线程安全，捕获 `IOException` 时自动将 `connected` 置为 `false` 实现断线检测。

通信序列：

```
RespCacheClient.execute("GET", "key")
    ↓
RespCodec.encode("GET", "key")  →  *2\r\n$3\r\nGET\r\n$3\r\nkey\r\n
    ↓
out.write(request)  →  发送到服务端
    ↓
RespCodec.decode(in)  →  从输入流读取响应
    ↓
返回 RespResponse 对象
```

#### 3.5.4 命令映射

| CacheServerClient 方法 | RESP 命令 | 备注 |
|---|---|---|
| `ping()` | `PING` | 返回 +PONG |
| `get(key)` | `GET key` | 返回批量字符串或 nil |
| `set(key, value, ttl)` | `SET key value [EX seconds]` | ttl>0 时附加 EX 参数 |
| `del(keys...)` | `DEL key1 key2 ...` | 返回整数（删除数量） |
| `scan(pattern)` | `SCAN [pattern]` | 非游标式，一次性返回 |
| `lpush(key, values...)` | `LPUSH key val1 val2 ...` | 返回整数（列表长度） |
| `rpush(key, values...)` | `RPUSH key val1 val2 ...` | 返回整数（列表长度） |
| `lpop(key)` | `LPOP key` | 返回批量字符串或 nil |
| `lrange(key, start, stop)` | `LRANGE key start stop` | 返回数组 |
| `ttl(key)` | `TTL key` | 返回整数（-2/-1/n） |

#### 3.5.5 WRONGTYPE 错误处理

`get()` 方法特殊处理服务端返回的 WRONGTYPE 错误。当 key 存在但类型不是 STRING（例如 LIST）时，服务端返回 `-WRONGTYPE` 或 `-wrong type`，`get()` 将其转换为 `Optional.empty()` 而非抛出异常。这使得 `CacheEntryLoader` 能够统一通过 `get()` 尝试读取 → 失败后通过 `lrange()` 回退，正确识别 LIST 类型条目。

```java
String upper = error.toUpperCase(Locale.ROOT);
if (upper.contains("WRONGTYPE") || upper.contains("WRONG TYPE")) {
    return Optional.empty();
}
```

兼容 Redis 标准格式（`-WRONGTYPE Operation...`）和第一组服务端格式（`wrong type`）。

#### 3.5.6 断线检测与自动重连

`execute()` 方法在发生 `IOException` 时自动标记 `connected = false`，实现所有命令的即时断线检测。`reconnect()` 方法先调用 `disconnect()` 关闭旧 Socket，再调用 `connect(lastHost, lastPort)` 创建新连接，避免连接泄漏。

```java
public boolean reconnect() {
    if (lastHost == null) return false;
    disconnect(); // 先关闭旧 socket
    connect(lastHost, lastPort); // 创建新连接
    return true;
}
```

#### 3.6.3 界面区域划分

- **区域1 — 连接管理栏**：服务器地址/端口输入、Connect/Disconnect/PING 按钮
- **区域2 — CRUD + List 操作栏**：Add/Update/Delete/Clear All 按钮，LPUSH/RPUSH/LPOP/LRANGE 操作，Rm Val（按值删除）、Rm Idx（按序号删除），`listLengthLabel` 显示当前 List 长度

#### 3.6.4 关键事件处理

所有涉及网络 I/O 的事件处理均通过 `runAsync()` 辅助方法在**后台线程**异步执行，执行完毕后通过 `setOnSucceeded` 在 JavaFX Application Thread 上更新 UI，避免界面卡顿。

| 事件 | 处理逻辑 |
|---|---|
| `onConnect()` | 先检查 `client.isConnected()`，已连接时弹出 WARNING 弹窗阻止重复连接；否则通过 `runAsync` 异步调用 `client.connect()`，成功后记录 `connectedHost/connectedPort`，启动 `Timeline` 心跳定时器，更新状态标签 |

| `onPing()` | 通过 `runAsync` 异步调用 `client.ping()`，结果显示在标签 |
| `onAdd()` | 加载 CacheEntryDialog.fxml 弹出对话框；通过 `runAsync` 异步调用 `client.set()` |
| `onDelete()` | 弹出确认对话框，通过 `runAsync` 异步调用 `client.del()` |
| `onClearAll()` | 弹出确认对话框，通过 `runAsync` 逐条异步删除 |
| `onSearch()` | 通过 `runAsync` 异步加载 `getAllEntries()`，用 `KeyPatternMatcher.filter()` 本地过滤 |
| `onShowAll()` | 清除搜索条件，调用 `refreshTableAsync()` 异步刷新 |
| `onLpush()` | 通过 `runAsync` 异步调用 `client.lpush()` + `client.lrange()`，刷新 List 面板 |

| `onLpop()` | 通过 `runAsync` 异步调用 `client.lpop()`，刷新显示 |
| `onLrange()` | 通过 `runAsync` 异步调用 `client.lrange()`，填充 ListView |

#### 3.6.6 异步执行辅助（runAsync）

```java
private <T> void runAsync(Callable<T> callable, Consumer<T> onSuccess, String errorTitle)
```

将网络操作封装为 `javafx.concurrent.Task` 在后台线程执行。`callable` 在后台线程运行（网络 I/O），`onSuccess` 在 JavaFX 线程回调（UI 更新），`errorTitle` 为空时不弹错误弹窗（用于心跳等后台任务）。

#### 3.6.7 心跳检测与自动重连

`Timeline` 定时器每 5 秒执行一次 `heartbeatTick()`：

```
heartbeatTick()
  ├─ ping 检测（异步）：isConnected() → ping() → 成功绿色 / 失败标记断开
  └─ 自动重连（异步）：!isConnected() → reconnect() → 成功更新状态标签
```

- 连接成功后通过 `startHeartbeat()` 启动，调用 `onDisconnect()` 时通过 `stopHeartbeat()` 停止
- 心跳使用 `errorTitle=null` 的 `runAsync`，失败时不弹错误对话框
- 服务端断线后心跳自动重连，重连成功后用户手动点 Refresh 加载数据

#### 3.6.8 数据加载策略（getAllEntries）

根据客户端类型采用不同加载策略：

| 客户端类型 | 策略 |
|---|---|
| `MockCacheClient` | 直接调用 `mock.getAllLocalEntries()` 读取本地内存存储 |

### 3.10 缓存条目加载模块（CacheEntryLoader）

**文件路径：** `src/main/java/com/cache/client/util/CacheEntryLoader.java`

#### 3.10.1 功能描述

RESP 模式下，通过 SCAN + GET + LRANGE 编排从服务端加载全部缓存条目。将底层 RESP 命令的差异（STRING 用 GET、LIST 用 LRANGE）统一为 `List<CacheEntry>` 返回。

#### 3.10.2 核心算法

```java
for (String key : client.scan("*")) {
    long ttl = client.ttl(key);           // 获取 TTL
    if (ttl == -2) continue;               // key 已过期，跳过

    Optional<String> value = client.get(key);  // 尝试 GET
    if (value.isPresent()) {
        entries.add(new CacheEntry(key, value.get(), ttl)); // STRING 类型
        continue;
    }

    List<String> values = client.lrange(key, 0, -1);  // fallback 到 LRANGE
    CacheEntry entry = new CacheEntry(key, CacheEntry.EntryType.LIST, ttl);
    entry.setListLength(values.size());
    entries.add(entry);                                  // LIST 类型
}
```

#### 3.10.3 类型识别策略

| 服务端返回 | `get()` 结果 | 后续操作 | 识别类型 |
|---|---|---|---|
| 正常返回字符串 | `Optional.of(value)` | 直接创建 CacheEntry | STRING |
| WRONGTYPE | `Optional.empty()` | 执行 `lrange()` 读取 | LIST |
| nil（不存在） | `Optional.empty()` | 执行 `lrange()`（同返回空列表） | LIST（空） |

### 3.11 Key 通配符匹配模块（KeyPatternMatcher）

**文件路径：** `src/main/java/com/cache/client/util/KeyPatternMatcher.java`

#### 3.11.1 功能描述

提供本地通配符匹配功能，用于主界面的搜索过滤。将 Redis 风格的 `*` 和 `?` 通配符 pattern 转换为 Java 正则表达式进行匹配，支持大小写不敏感。

#### 3.11.2 核心方法

| 方法 | 说明 |
|---|---|
| `matches(key, pattern)` | 判断 key 是否匹配 pattern，空 pattern 视为匹配全部 |
| `filter(entries, pattern)` | 过滤 CacheEntry 列表，返回匹配的条目（不修改原列表） |

#### 3.11.3 匹配规则

- 空字符串 / 空白 / `*` → 匹配所有 key
- `*` → 转换为 `.*`（匹配任意字符序列）
- `?` → 转换为 `.`（匹配任意单个字符）
- 非 `*` 和 `?` 字符 → 作为字面量（`Pattern.quote`），正则元字符不具特殊含义
- 匹配时统一转为小写（大小写不敏感）

## 6 错误处理

| 错误场景 | 错误类型 | 触发条件 | 处理策略 |
|---|---|---|---|
| 连接失败 | RuntimeException | Socket 连接异常（地址不可达/超时） | UI 层捕获后以红色标签显示错误信息 |
| 命令执行失败 | RuntimeException | RESP 服务端返回 `-ERR` | RespCacheClient 抛出，UI 层捕获显示 |
| RESP 解码错误 | IOException | 协议格式异常 / 网络断开 | RespCodec 抛出，RespCacheClient 包装为 RuntimeException |
| 连接断开 | IOException | 服务端关闭连接 / 网络故障 | 读流返回 -1，RespCodec 抛出 |
| 无效输入 | N/A | key/value 为空、端口格式错误 | UI 层前置校验，拒绝空操作 |
| 类型冲突（WRONGTYPE）| RuntimeException | STRING 命令操作 LIST key 或反之 | RespCacheClient.get() 返回 `Optional.empty()`，CacheEntryLoader fallback 到 LRANGE |
| 类型冲突（UI 层阻止） | N/A | onAdd() 检测到 key 已存在为 LIST 类型 | `isExistingListKey()` 弹出 WARNING 弹窗阻止覆盖 |
| List 操作失败 | RuntimeException | LPUSH/RPUSH/LPOP/LRANGE 执行错误 | `showError()` 弹出 ERROR 弹窗 |
| 重复连接 | N/A | 用户多次点击 Connect 按钮 | `onConnect()` 检查 `isConnected()`，已连接时弹出 WARNING 弹窗阻止 |
| 服务端断线 | IOException | 服务端关闭连接 / 网络故障 | `execute()` 捕获 IOException → `connected=false`；心跳 `ping()` 检测断开 → 自动 `reconnect()` 重连 |

### 6.1 错误传播路径

**RESP 命令错误路径：**
```
服务端返回 -ERR
    ↓
RespCodec 解析为 ERROR 类型 RespResponse
    ↓
RespCacheClient.expectOk() 检测 isError() → 抛 RuntimeException
    ↓
MainController 捕获异常 → statusLabel 显示错误信息（红色）
```

**WRONGTYPE 路径（静默处理）：**
```
服务端返回 -WRONGTYPE 或 -wrong type
    ↓
RespCodec 解析为 ERROR 类型 RespResponse
    ↓
RespCacheClient.get() 检测 error 包含 "WRONGTYPE" / "WRONG TYPE"
    ↓
返回 Optional.empty()（不抛异常）
    ↓
CacheEntryLoader 识别为 LIST 类型 → 通过 LRANGE 读取
```

**断线检测路径：**
```
服务端断开 / 网络故障
    ↓
execute() 读写抛出 IOException
    ↓
connected = false（RespCacheClient 自动标记）
    ↓
心跳 ping() 检测到断开
    ↓
心跳 reconnect() 自动重连（每 5 秒重试直到成功）
```

**UI 弹窗错误路径（List 操作）：**
```
服务端返回 -ERR
    ↓
RespCacheClient.lpush/rpush/lpop/lrange() 抛 RuntimeException
    ↓
MainController.onLpush/onRpush/onLpop/onLrange() 捕获异常
    ↓
showError() 弹出 Alert.ERROR 弹窗
```

### 6.2 TTL 错误码映射

| TTL 返回值 | 含义 |
|---|---|
| `n > 0` | 键存在，剩余 n 秒过期 |
| `-1` | 键存在但永不过期 |
| `-2` | 键不存在 |
